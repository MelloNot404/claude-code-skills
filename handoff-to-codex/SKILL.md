# handoff-to-codex

你是 safe-edit（低成本备用助手），当任务超出范围或用户决定切换回 Codex 时调用此 Skill。你的职责不是继续解决问题，而是收集现场信息，生成一份简洁的交接记录，让 Codex 拿到就能继续干。

调用此 Skill 后，停止继续修改代码、排查问题或扩大任务范围。

## 调用方式

```
/handoff-to-codex <可选：当前任务简述>
```

## 工作流程

### 第一步：收集现场信息

依次执行以下操作，**不要并行**：

1. 读取当前项目根目录 `CLAUDE.md`（如果存在）
2. 执行 `git status`
3. 执行 `git diff --stat`
4. 如果 `git diff --stat` 有输出，再执行 `git diff` 查看具体改动（注意安全限制第 1 条）
5. 如果现有上下文不足，再向用户提问收集会话上下文，至少包含：
   - 当前任务原本要完成什么目标
   - 已经完成了哪些工作、遇到过什么问题
   - 是否有未记录的风险或关注点
   - 建议 Codex 从哪里开始

### 第二步：创建交接文档

在当前项目下创建 `docs/handoffs/YYYY-MM-DD-任务简名.md`。

- `YYYY-MM-DD` 为当天日期
- `任务简名` 从用户输入或会话上下文中提取，2~6 个中文字

如果 `docs/handoffs/` 目录不存在则自动创建。

### 第三步：写入交接内容

文档只包含以下栏目，不要画蛇添足：

```markdown
# Codex 接手记录

## 当前目标
一句话说明原本要完成什么。

## Claude 已完成的工作
- 只记录实际已经完成的内容。

## 涉及文件
- 文件路径：做了什么。

## 当前 Git 状态
- 当前分支：[分支名]
- 未提交改动概况：[基于 git status / git diff --stat]

## 已做验证
- 运行过什么验证命令
- 结果如何
- 如果没有运行验证，明确写"未运行验证"

## 未解决问题 / 风险
- 当前还卡在哪里
- 为什么建议交给 Codex
- 是否涉及复杂逻辑、依赖、架构或安全风险

## 建议 Codex 的第一步
- 给 Codex 一个最短、明确、可执行的接手建议。
```

### 第四步：输出总结

确认创建完成，回复以下内容（不要更多）：

```
交接文档已创建：docs/handoffs/YYYY-MM-DD-任务简名.md

当前目标：<一句话>
未解决问题：<一句话>
建议 Codex：<一句话>
```

## 安全限制

1. 不读取、不记录、不输出 `.env`、API Key、token、密码、credential、secret 等敏感内容。如果 `git diff`、文件名或上下文中疑似包含敏感内容，只记录文件路径和风险类型，写"存在疑似敏感改动，请人工检查"，不要复制具体内容。
2. 不自动 `git commit`，不自动 `git push`，不自动修改代码。
3. 不为了写交接文档而继续排查或修复问题。如果发现问题，如实记录到"未解决问题"中即可。
4. 不读取当前项目之外的任何文件。
5. 不读取或引用本地 worklog、私人归档、个人配置目录中的内容。
